### 8.3 New finding: U4 waiting-banner does not clear

**Severity:** medium (cosmetic — banner stays after data flow is
clearly visible elsewhere on the page; no data path is broken).

**Symptom:** After Start, U4 banner appears as expected. As batches
arrive (verified by 22+ non-empty files in `kraken2/<sample>/batch_reports/`,
13 in `seqkit/`, plus active counter growth in the Dashboard tiles)
the banner does not auto-clear and continues to display
"elapsed: 3m 12s" while the pipeline is producing results.

**Co-symptom on the same pass:** the right-edge live indicator
remained pinned at the dashboard-load value `Last data: 06:20:29`
for the entire 4 m run, and the orange "Data may be stale" badge
stayed visible. Verdict-banner `Last updated` field DID update
correctly (06:23:20, 06:24:20, etc), and the metric tiles updated.

**Root-cause hypothesis (not yet confirmed):** the `compute_results_fingerprint`
callback (`callbacks.py:296-336`) appears to PreventUpdate for the
duration of the run. Both the `first_batch_seen` flag and the
`ts` field live in the same store payload, so a stuck fingerprint
freezes both the U4 clear and the F4 live-indicator. The store-update
helper `check_data_freshness` (`loader_utils.py:325`) walks the
correct subdirs and `_get_dir_latest_mtime` uses recursive `os.walk`
(May 7 obs 604 fix), so the underlying mtime walk is sound. The
freeze is more likely in how the callback's Inputs are wired
(perhaps `app-config` doesn't trigger or `update-interval` falls
under a debounce in this code path) or in `resolve_analysis_directory`
silently redirecting `main_dir` away from the run outdir. Because
metric tiles populate (they take `update-interval` as a backstop),
the data path itself is fine; only the fingerprint store is stuck.

**Repro:** GUI-launched run with `processing_mode: realtime`,
`sample_handling: by_barcode`, `pipeline_source: /path/to/local/nanometanf`,
warm conda cache. Click Start, feed `_watch/` via nanorunner. Watch
the U4 banner counter climb past 90 s while metric tiles populate.

**Workaround for operators:** the metric tiles ARE the source of
truth on a real run — operators can ignore the banner once tiles
report non-zero values.

**Suggested fix path:** instrument
`compute_results_fingerprint` with debug logging on each Input fire
and on each PreventUpdate branch; verify the resolved `main_dir`
matches the runtime outdir; verify Inputs include `update-interval`
as a backstop (per the CLAUDE.md "polling-tick backstop" pattern).
File as a P2 for the next iteration.
